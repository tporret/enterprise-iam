# Okta OIDC and SCIM Setup Guide

This guide configures Okta for:
- OIDC login (Authorization Code + PKCE)
- SCIM 2.0 provisioning

## Prerequisites

- Okta admin access.
- Your Enterprise IAM tenant URL in this format: `https://[tenant].yoursaas.com/`.
- An OIDC IdP configuration slot in Enterprise IAM.
- A generated SCIM bearer token in Enterprise IAM (admin side).

## Part A: OIDC Setup (Authorization Code with PKCE)

## 1. Create the Okta OIDC app

1. In Okta Admin, go to **Applications**.
2. Select **Create App Integration**.
3. Choose:
   - Sign-in method: **OIDC - OpenID Connect**
   - Application type: **Web Application**
4. Continue.

## 2. Configure sign-in redirect URI

Set this exact redirect URI in Okta:

- `https://[tenant].yoursaas.com/wp-json/enterprise-auth/v1/oidc/callback`

Enterprise IAM callback route is fixed to `/wp-json/enterprise-auth/v1/oidc/callback`.

## 3. Configure grant and scopes

Enterprise IAM sends these OIDC authorization parameters:

- `response_type=code`
- `scope=openid email profile`
- `code_challenge_method=S256`
- PKCE challenge and verifier values (generated per login)
- `state` and `nonce` correlation values

In Okta, allow Authorization Code flow and ensure user claims include email/profile values.

## 4. Copy Okta endpoints and credentials into Enterprise IAM

From Okta app and authorization server metadata, configure:

- `client_id`
- `client_secret`
- `issuer`
- `authorization_endpoint`
- `token_endpoint`
- `userinfo_endpoint`
- `jwks_uri`

Important runtime validation requirements in Enterprise IAM:
- URLs must be valid `https://` URLs.
- Endpoints must resolve to public, non-private addresses.

## 5. OIDC claims expected for provisioning

Enterprise IAM default claim keys:

- Email: `email`
- First name: `given_name`
- Last name: `family_name`
- Groups (optional for role mapping): `groups`
- Immutable user ID: `sub`
- Issuer: `iss`
- Email verification gate: `email_verified`

If attribute override mapping is enabled in your IdP config, custom claim keys are used instead.

## Part B: SCIM 2.0 Setup

## 1. SCIM base URL

Use this SCIM base URL in Okta provisioning:

- `https://[tenant].yoursaas.com/wp-json/enterprise-auth/v1/scim/v2/`

## 2. Authentication in Okta

Enterprise IAM expects HTTP Bearer authentication:

- Header: `Authorization: Bearer <token>`

Where to paste in Okta:

1. Open app provisioning settings.
2. Enable SCIM provisioning.
3. Set authentication mode to HTTP Header / Bearer token.
4. Paste the plaintext token generated by Enterprise IAM admin settings.

Enterprise IAM stores only the bcrypt hash server-side and validates incoming bearer tokens against it.

## 3. SCIM user attributes used

Enterprise IAM user provisioning expects SCIM Core User fields:

- `userName` (required, must be a valid email)
- `name.givenName`
- `name.familyName`
- `externalId`
- `active`

## 4. Supported SCIM actions

Supported:

- Create user: `POST /Users`
- Read user/list users: `GET /Users`, `GET /Users/{id}`
- Update user (full): `PUT /Users/{id}`
- Update user (partial): `PATCH /Users/{id}`
  - `active=false` suspends/deactivates account access
  - `active=true` reactivates account
- Deprovision user: `DELETE /Users/{id}`
  - Returns `204` on success

Group endpoints are also implemented:

- `GET /Groups`
- `POST /Groups`
- `PATCH /Groups/{id}`

## 5. Delete and suspend behavior notes

- PATCH with `active=false` is suspension/deactivation (sessions destroyed, account flagged suspended).
- DELETE is implemented and does not return 501 in current code:
  - Single-site: user can be deleted with reassignment logic.
  - Multisite: user is removed from site/network memberships per scope and may be globally suspended.

## 6. Test checklist

1. Test OIDC sign-in from Okta app.
2. Confirm callback returns to app and login succeeds.
3. Run Okta provisioning test:
   - Create user
   - Update profile
   - Deactivate (suspend)
   - Reactivate
   - Deprovision (delete)
4. Validate resulting WordPress user state and roles.
